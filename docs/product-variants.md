# واریانت محصول (تسک ۱۶)

این سند توضیح می‌دهد واریانت‌های محصول (مثل سایز و رنگ) چطور در نیشا کار می‌کنند و برای راه‌اندازی چه کارهایی لازم است.

## راه‌اندازی

یک migration جدید اضافه شده است. بعد از pull حتما اجرا کنید:

```bash
cd backend
alembic upgrade head
```

نسخه migration: `20260706_0015_product_variants` (بعد از `20260706_0014_discount_codes`).

## مدل داده (بک‌اند)

- جدول جدید **`product_variants`**: هر ردیف یک واریانت از یک محصول است.
  - `name` — نام واریانت (مثلا «سایز L»)
  - `price_override` — قیمت اختصاصی (اختیاری؛ اگر خالی باشد قیمت خود محصول استفاده می‌شود)
  - `stock_quantity` — موجودی همان واریانت
  - `sort_order` — ترتیب نمایش
  - `is_active` — فعال/غیرفعال
- جدول `order_items` دو ستون جدید دارد:
  - `variant_id` — ارجاع به واریانت (با حذف واریانت، NULL می‌شود تا سفارش‌های قدیمی سالم بمانند)
  - `variant_name_snapshot` — نام واریانت در لحظه ثبت سفارش (برای فاکتور)

## قواعد مهم

1. **موجودی محصولِ دارای واریانت خودکار است**: وقتی محصولی واریانت فعال دارد، موجودی کل محصول برابر جمع موجودی واریانت‌های فعال نگه داشته می‌شود (در ذخیره محصول و در هر خرید/لغو سفارش همگام می‌شود).
2. **انتخاب واریانت اجباری است**: اگر محصولی واریانت فعال داشته باشد، سفارش بدون `variant_id` با خطای 422 رد می‌شود («انتخاب واریانت برای «محصول» الزامی است»). برعکس، ارسال واریانت برای محصول بدون واریانت هم 422 می‌دهد.
3. **قیمت خط سفارش**: `price_override` واریانت اگر تنظیم شده باشد، وگرنه قیمت محصول.
4. **برگشت موجودی**: با لغو سفارش، موجودی واریانت و موجودی والد هر دو برمی‌گردند.

## API

- **فروشنده** — `POST/PUT /api/v1/seller/products`: فیلد اختیاری `variants` (آرایه‌ای از `{name, price_override?, stock_quantity?, sort_order?, is_active?}`). ارسال آرایه، کل واریانت‌ها را جایگزین می‌کند؛ `null` یعنی بدون تغییر ساختار قبلی حذف شوند/نباشند.
- **فروشگاه عمومی** — `GET /api/v1/public/stores/{slug}`: هر محصول `variants` (فقط فعال‌ها) را برمی‌گرداند.
- **ثبت سفارش** — آیتم‌های سفارش می‌توانند `variant_id` داشته باشند. آیتم‌های تکراری با کلید (محصول، واریانت) ادغام می‌شوند.
- **پیگیری سفارش/فاکتور** — هر آیتم `variant_name` دارد.

## فرانت‌اند

- **فرم محصول فروشنده** (`components/seller/ProductForm.tsx`): بخش «واریانت‌ها» برای افزودن/حذف/مرتب‌سازی واریانت با نام، قیمت اختصاصی، موجودی و وضعیت فعال.
- **صفحه محصول** (`ProductPageClient.tsx`): دکمه‌های انتخاب واریانت؛ قیمت و موجودی بر اساس واریانت انتخابی؛ واریانت ناموجود غیرفعال نمایش داده می‌شود.
- **کارت محصول** (`ProductCard.tsx`): برای محصول واریانت‌دار دکمه «انتخاب واریانت» به صفحه محصول می‌برد.
- **سبد خرید** (`CartContext.tsx` / `CartDrawer.tsx`): هر خط سبد با (محصول + واریانت) شناسایی می‌شود؛ نام واریانت زیر عنوان نمایش داده می‌شود؛ هنگام همگام‌سازی با فروشگاه، خط‌های واریانتِ حذف‌شده/ناموجود پاک می‌شوند.
- **پرداخت** (`app/store/[slug]/checkout/page.tsx`): `variant_id` هر خط در payload ارسال می‌شود و نام واریانت در خلاصه سفارش دیده می‌شود.
- **فاکتور و پیگیری** (`InvoiceView.tsx` / `OrderTrackDetails.tsx`): نام واریانت کنار نام محصول.

```bash
cd backend

```

## نکات

- سبدهای خرید قدیمی (بدون واریانت) به‌صورت خودکار پاک‌سازی می‌شوند اگر محصول حالا واریانت داشته باشد.
- حذف واریانت، سفارش‌های قبلی را خراب نمی‌کند (snapshot نام و NULL شدن ارجاع).
